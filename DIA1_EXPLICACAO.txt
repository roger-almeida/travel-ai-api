3. Trip.java — O Modelo (Model)
   Caminho: src/main/java/com/travel/model/Trip.java

   @Data             → gera: getters, setters, toString, equals, hashCode
   @NoArgsConstructor → gera: public Trip() {}
   @AllArgsConstructor → gera: public Trip(String id, String destination, ...)
   public class Trip {
       private String id;
       private String destination;   // destino da viagem
       private String traveler;      // nome do viajante
       private LocalDate startDate;  // data de ida
       private LocalDate endDate;    // data de volta
       private double budget;        // orçamento em USD
       private String status;        // PLANNED, IN_PROGRESS, COMPLETED
   }

   CONCEITO — Lombok:
   Sem Lombok você escreveria ~50 linhas de getters/setters manualmente.
   Com @Data, 1 anotação faz tudo. O compilador gera o código no build.

   CONCEITO — Factory Method (padrão OOP):
   O método estático Trip.create(...) encapsula a criação do objeto,
   garantindo que o ID (UUID) e o status ("PLANNED") sempre sejam
   gerados corretamente. Quem cria uma Trip de fora não precisa saber
   desses detalhes.

   Por que UUID?
   UUID.randomUUID() gera um identificador único universal (ex: "a3f2-...").
   Garante IDs únicos sem precisar de banco de dados sequencial.

4. TripService.java — Camada de Negócio (Service)
   Caminho: src/main/java/com/travel/service/TripService.java

   @Service
   public class TripService {
       private final List<Trip> trips = new ArrayList<>(); // banco em memória

       public List<Trip> findAll() { ... }
       public Optional<Trip> findById(String id) { ... }
       public Trip create(...) { ... }
   }

   CONCEITO — @Service e Injeção de Dependência:
   Você nunca vai chamar "new TripService()" no código.
   O Spring cria e gerencia uma instância única (Singleton) e a injeta
   onde for preciso via @Autowired. Isso se chama IoC (Inversion of Control).

   CONCEITO — Optional<Trip>:
   Em vez de retornar null (que causa NullPointerException se você esquecer
   de verificar), o Optional força quem usa o método a tratar o caso "não encontrado".
   É um contrato explícito: "pode ou não ter um valor".

   CONCEITO — List em memória (temporário):
   Por enquanto os dados somem ao reiniciar o app.
   Na Semana 2 vamos substituir pela AWS DynamoDB — o código do Service
   não precisará mudar graças à separação de camadas.

5. TripController.java — Endpoints HTTP (Controller)
   Caminho: src/main/java/com/travel/controller/TripController.java

   @RestController         → responde requisições HTTP e retorna JSON
   @RequestMapping("/trips") → todas as rotas começam com /trips
   public class TripController {

       @Autowired
       private TripService tripService;

       @GetMapping              → GET /trips         → lista todas
       @GetMapping("/{id}")     → GET /trips/{id}    → busca por ID
       @PostMapping             → POST /trips        → cria nova viagem
   }

   ENDPOINTS:
   ┌────────┬──────────────┬───────────────────────────┬──────────────────┐
   │ Método │ Rota         │ O que faz                 │ Retorno HTTP     │
   ├────────┼──────────────┼───────────────────────────┼──────────────────┤
   │ GET    │ /trips       │ Lista todas as viagens    │ 200 OK + array   │
   │ GET    │ /trips/{id}  │ Busca viagem por ID       │ 200 OK ou 404    │
   │ POST   │ /trips       │ Cria nova viagem          │ 201 Created      │
   └────────┴──────────────┴───────────────────────────┴──────────────────┘

   CONCEITO — @PathVariable vs @RequestBody:
   @PathVariable: extrai valor da URL   → GET /trips/abc123  (id = "abc123")
   @RequestBody:  lê o JSON do body     → POST com {"destination": "SP", ...}

   CONCEITO — ResponseEntity:
   Permite controlar o status HTTP de resposta.
   ResponseEntity.ok(body)           → 200 OK
   ResponseEntity.notFound().build() → 404 Not Found
   ResponseEntity.status(CREATED)    → 201 Created

--------------------------------------------------------------------------------
ARQUITETURA EM CAMADAS
--------------------------------------------------------------------------------

   Requisição HTTP
         ↓
    Controller    ← recebe JSON, valida formato, delega ao Service
         ↓
     Service      ← regras de negócio (validações, lógica)
         ↓
   Repository     ← acesso a dados (hoje: List / Semana 2: DynamoDB)
         ↓
      Model       ← representa os dados (Trip)

   REGRA DE OURO: cada camada só conhece a camada abaixo dela.
   Controller conhece Service. Service conhece Repository. Ninguém "pula" camadas.

   BENEFÍCIO: quando trocar o banco de memória pelo DynamoDB, só muda
   o Repository — Controller e Service ficam intocados.

--------------------------------------------------------------------------------
DIFERENCA ENTRE .java E .class
--------------------------------------------------------------------------------

   Voce vai encontrar dois arquivos com o mesmo nome mas extensoes diferentes:
   Trip.java e Trip.class, TripService.java e TripService.class, etc.

   .java  → arquivo FONTE — voce escreve e edita. Texto puro, legivel.
            Fica em: src/main/java/com/travel/
   .class → arquivo COMPILADO — a JVM executa. Binario, nao legivel.
            Fica em: target/classes/com/travel/

   O fluxo:

   Voce escreve        Compilador Java          JVM executa
     Trip.java   ─────────────────────►   Trip.class
   (texto legivel)   (javac / Maven)      (bytecode binario)

   ANALOGIA: o .java e uma receita escrita em portugues. O .class e essa
   receita traduzida para uma linguagem que o "forno" (JVM) entende e executa.

   REGRAS PRATICAS:
   - Nunca edite um .class — qualquer mudanca some na proxima compilacao
   - A pasta target/ inteira e descartavel (pode deletar a qualquer hora)
   - target/ nao vai para o GitHub (esta no .gitignore) — so src/ vai
   - Ao rodar "./mvnw spring-boot:run", Maven compila .java → .class → JVM roda

--------------------------------------------------------------------------------
O QUE E UMA INSTANCIA
--------------------------------------------------------------------------------

   Uma CLASSE e um molde (planta baixa de uma casa).
   Uma INSTANCIA e o objeto construido de verdade na memoria RAM.

   // Trip e a CLASSE — so existe no codigo, e o molde
   public class Trip {
       private String destination;
       ...
   }

   // trip1 e trip2 sao INSTANCIAS — objetos reais criados na memoria
   Trip trip1 = new Trip();   // construiu uma "casa" na memoria
   Trip trip2 = new Trip();   // construiu outra "casa" independente

   trip1 e trip2 sao objetos separados. Mudar o destination de trip1
   nao afeta trip2. Eles existem na RAM enquanto o programa roda.

   Instanciar = criar um objeto real a partir de um molde usando "new".

--------------------------------------------------------------------------------
O QUE E @Autowired E INJECAO DE DEPENDENCIA
--------------------------------------------------------------------------------

   O PROBLEMA sem Spring:
   Para o TripController usar o TripService, voce criaria manualmente:

   public class TripController {
       private TripService tripService = new TripService();  // voce constroi
   }

   Isso cria ACOPLAMENTO: o Controller e responsavel por saber como construir
   o Service. Se o Service mudar, o Controller precisa mudar junto.

   O QUE O SPRING FAZ:
   Quando o app sobe, o Spring varre o codigo, encontra tudo marcado com
   @Service, @RestController etc., cria uma instancia de cada um e guarda
   num "deposito" chamado Container:

   Spring Container (deposito de objetos prontos)
   +------------------------------------------------+
   |  TripService    → instancia criada e guardada  |
   |  TripController → instancia criada e guardada  |
   +------------------------------------------------+

   Quando voce usa @Autowired:

   @RestController
   public class TripController {
       @Autowired
       private TripService tripService;  // "Spring, me entrega a instancia pronta"
   }

   Voce esta dizendo: "nao vou criar isso — voce me entrega o objeto que ja
   esta no seu container". O Spring coloca o objeto nessa variavel pra voce.

   ANALOGIA DO RESTAURANTE:
   - Sem Spring: o garcom vai a fazenda, planta o tomate, cozinha e serve.
   - Com @Autowired: o garcom pede para a cozinha e recebe o prato pronto.
   O TripController e o garcom. O TripService e o prato. O Spring e a cozinha.

   POR QUE ISSO IMPORTA:

   Situacao                     Sem DI              Com @Autowired
   ------------------------------------------------------------------
   Trocar implementacao         Muda em todo lugar  Muda so a classe
   Testar Controller isolado    Dificil             Injeta Service falso
   Multiplos usos do Service    Cria N instancias   Spring cria 1, reutiliza

   Esse ultimo ponto e o padrao SINGLETON: por padrao o Spring cria UMA
   unica instancia de cada classe e a compartilha em todo o app.
   Economiza memoria e garante consistencia.

   RESUMO EM UMA FRASE:
   @Autowired = "Spring, voce cria e gerencia esse objeto — so me entrega
   quando eu precisar", em vez de voce usar new e carregar essa responsabilidade.
